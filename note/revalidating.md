# 🧠 乐观更新与 Next.js 数据流思维导图

## 1. 核心概念 (Concept)

* **什么是乐观更新**：在数据尚未真正写入数据库（未落库）前，前端本地先行更新 UI，给用户“瞬间成功”的体验。
* **典型场景**：点赞按钮、信息发送、Todo 任务添加。

## 2. 方案一：React Query (SPA 异步 JSON 模式)

* **适用场景**：传统单页应用，或 Next.js 中配合 Route Handlers (API 路由) 使用。
* **运行特点**：纯客户端异步流，收发轻量 JSON。不占主线程，**天然不需要 transition 配合**。
* **Todo 实现核心三步走（解密）**：
* **① `onMutate` (偷跑)**：点击瞬间拦截请求，手动修改本地内存缓存（直接把新 Todo 塞进列表）。
* **② `onError` (回滚)**：如果网络报错，利用闭包中备份的旧数据，自动覆盖回滚。
* **③ `onSettled` (对齐)**：无论成功或失败，调用 `invalidateQueries` 作废本地缓存，强制从服务器拉取最终的真理数据。



## 3. 方案二：useOptimistic + useTransition (Next.js 全栈全量重洗模式)

* **适用场景**：重度依赖 Server Components (RSC) 与 Server Actions 的现代 Next.js 架构。
* **useOptimistic (临时演员)**
* **作用**：专用于快捷管理过渡期的临时乐观状态。
* **合体**：Server Action 成功并带来全新真实 Props 时，临时状态退场，真Props无缝接管。
* **自动回滚**：**完美依赖 Transition 的生命周期**，一旦异步 Action 报错，沙盒崩溃，界面自动回退。


* **useTransition (防卡死刹车)**
* **为什么需要**：Server Action 触发后，服务器会自上而下重新跑完当前整条路由线，带回庞大的 RSC 组件数据。浏览器如果强行同步合并，会卡死主线程，阻塞用户输入。
* **如何解决**：利用 **React 18 并发渲染 (Concurrent Rendering)** 开启时间切片，将重度重绘标记为“低优先级”，主线程优先响应用户的打字、点击。


* **底层运行真相 (Fiber 机制校正)**
* **服务端（大包大揽）**：服务器**必须重跑一遍**当前页面的整条组件树（地图炮），全量生成 RSC Payload。
* **客户端（微创手术）**：客户端 React 拿到整页数据后，通过 **Fiber 架构进行虚拟 DOM 逐层 Diff（比对）**。它明确知道哪个小组件变了，只做局部数据覆盖，**绝不白屏，也不丢失任何状态**。



## 4. 辅助工具：router.refresh()

* **作用**：显式通知服务器：“老家数据变了，把我（当前页面）所有的服务端组件全量重新渲染一遍”。
* **特性**：
* **局部覆盖**：数据变了就刷，不白屏，**完美保留客户端的所有 React 状态**（如 `useState`、滚动位置、未提交的表单输入）。


* **注意点（巨大误区）**：它只是重新跑一遍组件代码，**无法清除服务器的硬缓存**（如 `use cache`）。如果服务端数据被锁死，必须在服务端先调用 `revalidatePath/Tag`。

### 💡 核心 Review 备注：

1. **React Query 部分**：帮你补充了 `onMutate` -> `onError` -> `onSettled` 的标准乐观更新三步走，这是 React Query 应对 Todo 的官方标准解。
2. **底层真相校正**：原思维导图中 *“react 页面组件树不用重跑一遍”* 容易产生歧义。修正为：**服务端必须重跑组件树生成数据，但客户端 React 靠 Fiber 机制做微创手术，不需要重新挂载，因此用户感知不到整页刷新**。
