Suspense
服务端响应速度是可控的
首屏渲染时间 = 网络建连与传输耗时 + 后端服务响应耗时 + 前端关键资源（HTML/CSS/JS）下载与解析渲染耗时
 服务器的复杂逻辑：复杂计算、耗时操作必须是异步的，否则，页面加工会被阻塞过长时间，比如todos、头部、底部都已经渲染完毕，列表需要去请求以便动态渲染。服务器可以先将已经确定的页面返回出去，列表数据源来了加工成RSC返回给客户端。

服务器处理复杂计算或耗时操作时，必须采用异步架构，否则会导致页面加工受阻、白屏时间过长。
以待办事项（Todos）页面为例，服务器应采用流式渲染（Streaming）或渐进式渲染策略：优先将头部、底部等静态、已确定的 HTML 结构首批返回给浏览器；当列表数据源加载完成后，再在服务端将其动态加工为 React 服务器组件（RSC）数据流，追加发送至客户端进行局部渲染。

Suspense：前端使用SuspenseTODO的List展示先用Loading效果占位，服务端异步逻辑执行完毕，List列表在渲染出来。如果RSC都是动态计算，那么建议使用Suspense先展示loading在展示ui
